1. 后端分层架构

后端代码怎么组织的?

一个请求像流水线一样从上往下穿过各层,每层只干一件事:

HTTP 请求
  → routes/        路由表:URL → 哪个函数      (像 switch-case 分发)
  → middlewares/   中间件:鉴权、错误捕获       (请求的前置拦截器)
  → controllers/   控制器:收参数、调 service、返回响应(不写业务逻辑)
  → services/      业务逻辑:状态机、事务、缓存编排(核心)
  → repositories/  数据访问层(DAO):只写 SQL
  → loaders/       基础设施:MySQL 连接池、Redis 客户端
  → MySQL / Redis
C++ 类比:就是"接口层 / 逻辑层 / 数据层"分离。换数据库只改 repository,改业务只改 service,互不影响。
请求穿过 路由→中间件→控制器→服务→仓库→数据库,每层职责单一。

2. JS / Node 与 C++

JS 是后端语言吗?Node 是什么?

名词是什么C++ 类比
JavaScript语言本身C++ 语法规则
Node.js能在服务器跑 JS 的运行时(底层 C++ 写的 V8 引擎 + 文件/网络/DB 能力)运行环境(编译目标)
ExpressWeb 框架(一个库)第三方库

JS 出身是前端语言(浏览器里让网页动),2009 年 Node.js 让它也能写后端。所以它是"前后端通用",不是单纯的后端语言。同一份 JS,换运行环境(浏览器 / Node)就能干不同的活。

C++ 类比:JS 是语言;浏览器和 Node 是两个不同"运行环境",像同一份 C++ 代码分别编译成 Windows 程序和 Linux 程序。require('mysql2') 这类连数据库的能力来自 Node 环境,不是语言自带。

3. async / await

await 和 C++ 函数调用有什么不一样?前后 await 会顺序执行,那怎么和别的请求异步?

根本区别:阻塞+多线程 vs 单线程+事件循环

顺序执行?等待时线程在干嘛
C++ 阻塞调用是死占着线程空转,别人用不了
JS await是(一样顺序)线程离开去服务别的请求,结果回来再继续

await 不阻塞线程,它只"暂停当前这个函数",把唯一的线程让给别人。等的是一个 Promise(未来值的"提货单",类比 std::future)。

const x = await query();   // 线程:把请求发出去(几微秒)
                           // ↓ 等数据库的几十毫秒,线程已离开去干别的
console.log(x);            // 结果回来,线程回来从这继续

"顺序"与"异步"为何不矛盾

"顺序"约束的是你函数内部的代码行;异步发生在每个 await 等待 IO 的空档——那段时间线程跑去执行了别的请求。

生活类比:C++ 式 = 炖汤时盯着锅站 20 分钟啥也不干;await 式 = 汤下锅后立刻去切别的菜(处理别的请求),汤好了再回来盛。步骤顺序没变,但没干等。
await 暂停当前函数、让出线程;该等的地方忘写 await,会拿到"提货单"(Promise)而非真实数据 —— 头号坑。

4. 单线程 / 多进程 / 多核

Node 单线程,在多核机器上和单核不是一样?在哪创建多进程?多服务器怎么保证数据库一致?

一个 Node 进程基本只用一个核。要吃满多核 → 按核数多开几个 Node 进程(cluster / PM2 / 多容器),OS 把它们铺到各核。

# PM2:零代码改动,按核数起满,自动共享端口
pm2 start src/server.js -i max --name miniapp

扩展链:单线程 → 多进程 → 多机器,靠同一个原则

访问变多:  多 App ── 1 MySQL ── 1 Redis   (加 App + 负载均衡,一致性"免费")
读压力大:  多 App ── 主库+从库 ── Redis    (读写分离,接受复制延迟)
写也扛不住: 多 App ── 分库分表/TiDB ── Redis (最后才上,跨库事务最棘手)
关键:进程之间不共享内存(不同于 C++ 多线程)。能水平扩展的前提是"应用无状态"——会话放 Redis、数据放 MySQL,多个进程读同一份共享数据,所以天然一致。把状态从应用里挤出去,是后面一切扩展能力的地基。

5. 抗并发(并发 vs 并行)

Node 单线程,抗并发不是很差吗?

恰恰相反——对 Web 服务(99% 时间在等 IO),单线程事件循环是抗高并发的最优模型之一。要分清两个词:

词含义Node
高并发连接同时扛几万个连接(各自大多在等)极强 ✅
CPU 算力用多核做大量计算弱 ❌(单核)

C++ thread-per-connection:1 万连接 ≈ 1 万线程,吃内存、上下文切换累死(C10K 问题)。Node:连接在等待时不占线程,一个线程照看海量连接。Nginx 也是这个模型。

Node 抗"高并发 IO 连接"极强,弱在"CPU 密集计算"——一段长时间不 await 的重活会卡死整条线。

6. 数据库与"加字段"

迭代版本要给数据库加记录信息怎么办?怎么更新数据库?迭代期间会不会不可用?

数据库是一个"一直在跑的程序"

MySQL 是常驻进程,你给它发一句 SQL 文本,它立刻执行、立刻改硬盘数据。没有编译、不用重启。

# 三种发命令方式,本质一样:
mysql -u root -p           # ① 交互窗口手敲
mysql -u root -p db < x.sql # ② 把 SQL 文件喂进去
# ③ 图形工具(DBeaver/Navicat)点执行

加字段 = 迁移(migration)

不要改 init.sql(那是"从零建库"用的,且只在数据卷为空时跑一次)。写一个迁移文件:

ALTER TABLE users ADD COLUMN phone VARCHAR(20) NOT NULL DEFAULT '';

然后顺着分层往上改:repository 的 SELECT/UPDATE 带上新列 → service 业务规则(注意清/等 Redis 旧缓存)→ controller 收新参数。

迭代期间会不会停服?

改动影响可用性做法
加列(可空/有默认值)不影响先迁移、后发代码,一步搞定
删列/改名/改类型直接做会挂扩展-收缩三步走,绝不一步到位
换代码进程单进程会断几秒多实例滚动部署 + 优雅关闭
C++ 类比:C++ 改 struct 加成员重新编译即可;但数据库是持久的,线上已有真实数据,必须用 ALTER 在不丢数据的前提下"追加一列"。ON UPDATE CURRENT_TIMESTAMP 那种由数据库自己维护,不用代码管。
加列向后兼容("先 DB 后代码"+ 默认值)即可不停机;删/改是破坏性变更,必须拆成几个版本慢慢来。

7. Redis 的两类用途

Redis 在项目里干嘛?

内存数据库,超快但断电丢(只放"丢了能重建"的)。看 key 前缀区分两类用途:

用途key失效
会话(登录态)session:{userId}7 天 TTL / 登出删
缓存(Cache-Aside)user:profile:{id} / order:detail:{id}TTL(带抖动)/ 更新刷新
验证码(本项目邮箱认证)verify:register:{email}5 分钟 TTL

缓存三大坑的对策:防穿透(把"不存在"也缓存短 TTL)、防雪崩(TTL 加 ±10% 抖动)、写后刷新(更新库后直接写回最新值)。

8. Docker:卷 / 数据文件

init.sql 不是脚本怎么跑起来的?数据库文件在哪?敏感数据都放卷里吗?

init.sql 怎么"自动跑"

compose 里 ./sql/init.sql:/docker-entrypoint-initdb.d/init.sql 只是把文件映射进容器某目录。MySQL 镜像的启动脚本约定:首次启动(数据目录为空)时,会把该目录下的 .sql 文件喂给数据库执行。等价于你手动 mysql < init.sql,只是镜像替你做了。非空就跳过 —— 这就是"改了 init.sql 重启不生效"的原因。

数据文件在哪

compose: mysql_data:/var/lib/mysql   ← named volume
主机真实路径: /var/lib/docker/volumes/<项目名>_mysql_data/_data
每张表一个文件: users.ibd / orders.ibd ...(InnoDB)
概念是什么持久?
容器正在跑的 MySQL 进程临时,删了重建无所谓
卷(volume)硬盘上的数据文件持久,真正的家当
C++ 类比:.ibd 文件就像你 C++ 程序写的二进制数据文件,只是格式由 InnoDB 定、由 MySQL 进程独占(不能用文本编辑器直接改)。
"持久数据进卷;秘密(密码/密钥)走环境变量/密钥管理而非卷;代码进镜像"。生产里数据库常独立部署,容器主要装无状态应用。

9. socket 通信与协议

JS 和数据库是 socket 连的吗?会转换成什么数据流?MySQL 监听 HTTP 吗?都走 socket 不会有效率问题?

是的——Node 和 MySQL/Redis 是独立进程,靠 TCP socket 通信,跟 C++ 连远程服务收发字节一样。

C++ 类比:getPool().query(sql, params) ≈ 封装好的 send(编码后的SQL) + recv(结果) + parse。"连接池"就是一组预先建好、循环复用的 socket。

TCP 是通道,协议是语言

端口谁说什么语言
3000Node(Express)HTTP(唯一说 HTTP 的地方)
3306MySQLMySQL 二进制协议(不是 HTTP)
6379RedisRESP 协议

发给数据库的真实数据流

代码里 WHERE id IN (?,?) + [1,2],mysql2 在发送前拼成最终文本,MySQL 实际收到:... WHERE id IN (1,2)。打成 length-prefixed 的字节包:

[包体长度 3B][序号 1B][命令 1B=0x03 COM_QUERY][SQL 文本字节...]
C++ 类比:就是"先发长度,再发类型,再发内容"的 length-prefixed 自定义协议,只不过格式由 MySQL 定。

效率:成本真实但可控

一次本机往返毫秒级(比进程内调用慢几万倍,但绝对值不大)。杀手是"往返次数太多"。对策全在项目里:连接池省建连、findByIds 批量把 N 次压成 1 次、Redis 缓存把"贵的 MySQL 往返"换成"便宜的内存往返"。

进程内存(纳秒) > Redis内存(微秒,隔socket) > MySQL磁盘(毫秒,隔socket+读盘)

11. 前端框架(Vue 全家桶)

前端框架怎么组织?Pinia store 那段什么意思?

前端是跑在浏览器里的 SPA(单页应用):只有一个 index.html,页面切换由 JS 换内容,不向服务器要新网页。

库角色类比
Vue3界面框架:数据变 UI 自动变GUI 框架(Qt)
Vite构建/开发服务器 + 代理编译器 + 构建系统
Vue RouterURL ↔ 显示哪个页面浏览器端的 routes/
Pinia跨页面共享状态(登录态)全局单例 / 共享状态
axios发 HTTP 请求"之于后端 API"的连接库

响应式:Vue 的精髓

ref 变量改了,界面自动重画;用户输入,变量自动更新。你不用手动操作 DOM。

Pinia store = 带响应式 + 持久化的全局单例

defineStore('auth', {
  state:   () => ({ token, user }),      // 数据(从 localStorage 恢复,刷新不丢)
  getters: { isLoggedIn: s => !!s.token },// 派生只读值(类比 const 成员函数)
  actions: { login(){...}, logout(){...} }// 改数据的方法,改 this.xxx → 界面自动更新
})

语法点:() => ({...}) 箭头返回对象要加圆括号;|| '' 兜底;!! 转布尔;JSON.parse/stringify 在对象↔字符串间转(localStorage 只能存字符串)。

Vue3 管"数据变界面变";Router 管页面切换;Pinia 是共享状态(类比会话);axios+api 层跟后端对话;Vite 是构建工具兼开发服务器,用代理把 /api 转给后端解决跨域。

12. 浏览器对象模型

浏览器里有哪些对象?我要操作和获取什么?(像 C++ 我知道 std::string 有哪些方法)

JS 里有两类对象,分清就不模糊了。

第一类:纯数据对象 — 和 C++ 容器一样

字符串/数组/对象,活在 JS 内存堆,调方法做纯计算。

"a@b.com".split("@")   // 类比 std::string 方法
[3,1,2].map(x=>x*2)    // 类比 std::vector
Object.keys({id:1})    // 取所有键

第二类:浏览器"句柄对象" — 像 C 的 OS 句柄

浏览器预先塞给你的全局对象,是通往真实世界(屏幕/存储/网络)的句柄。调它的方法,真实世界就变。

对象代表操作/获取什么
window全局根(整个标签页)一切都挂它下面
document页面 DOM(一棵对象树)getElementById 找元素;改元素 → 屏幕变;读元素 → 拿界面状态
DOM 元素每个 HTML 标签.value 读输入、.textContent 改文字、.addEventListener 监听点击
localStorage持久键值存储setItem/getItem/removeItem
fetch网络(通往后端)发 HTTP(axios 封装了它)
console控制台log(类比 printf)
C/C++ 类比:第一类 = std::string/std::vector(自己算);第二类 = OS 句柄 FILE*/socket/GUI 控件(操作它就影响真实世界)。DOM 就是浏览器版的 GUI 控件树:改 DOM 对象 → 浏览器自动重画屏幕。
Vue 把第二类藏起来了:项目里几乎不直接写 document.getElementById。你只改数据,Vue 在底层自动调 DOM API 更新屏幕(响应式)。底层有 DOM 这套句柄,但平时用 Vue 不直接碰。

14. HTML 标签速查 + SPA 空壳

dist/index.html 这个语法怎么看?

语法三条规则

<标签 属性="值">内容</标签>
   ↑开始标签              ↑结束标签(带斜杠)

① 标签成对(<title>...</title>),无内容的自闭合(<meta />);② 属性写在开始标签里(lang="zh");③ 标签嵌套成树 —— 这棵树就是 DOM。

C++ 类比:HTML 不是编程语言(无逻辑/循环),只是纯描述性的嵌套结构,类似 XML/YAML 配置。浏览器读它 → 建成 DOM 对象树 → 画到屏幕。

常用标签速查

标签作用
<!DOCTYPE html>声明 HTML5 文档,固定放第一行
<html>根标签,整页都在里面
<head>头部:元信息,不直接显示
<meta charset>字符编码(UTF-8 防乱码)/ viewport 移动端自适应
<title>浏览器标签页标题
<script src>加载并执行 JS 文件
<link rel="stylesheet">加载 CSS 样式表
<body>真正显示在页面上的内容
<div>通用容器块(分区)
<span>行内通用容器(包一小段文字)
<h1>~<h6>标题(大→小)
<p>段落
<a href>超链接
<img src>图片
<ul>/<ol>/<li>无序/有序列表 + 列表项
<form>表单容器
<input>输入框(type=text/password/email...)
<button>按钮
id / class 属性id 唯一标识;class 分类(CSS/JS 靠它们定位元素)

SPA 空壳:body 里几乎是空的

<body>
  <div id="app"></div>   <!-- 空容器,登录框/按钮 HTML 里一个都没有 -->
</body>

界面去哪了?由 <script> 加载的 Vue 应用在运行时动态生成,塞进 #app:

浏览器加载 index.html → 看到空的 <div id="app">
  → 执行 <script> 里的 Vue 代码
    → main.js: createApp(App).mount('#app')
      → Vue 把整个界面"画"进这个空 div → 登录框/按钮才出现
dist 是编译产物:源文件 body 里只有 <script src="/src/main.js">;npm run build 后被打包成带哈希名的 /assets/index-xxxx.js + 自动注入 CSS <link>。哈希名用于缓存:内容变→文件名变→强制重新下载。
HTML 用成对标签描述页面结构(非编程语言),浏览器读它建成 DOM 树;SPA 的 body 只是个空壳 <div id="app">,真正界面由 JS(Vue)运行时生成塞进去。
